# CleanOps

전국 단위 세탁 프랜차이즈 기업의 **내부 통합 운영 시스템**을 실제 운영 가능한 수준으로 설계한 프로젝트입니다.

CleanOps는 단순 CRUD 포트폴리오가 아니라, **매장 접수 → 공장 처리 → 물류 → 출고 → 환불/클레임 → 정책 통제 → 정산 → 감사**까지 이어지는 전체 운영 흐름을 하나의 플랫폼으로 설계하는 것을 목표로 합니다.

> 현재 상태: **Phase3 설계 산출물 작성 완료 / 조건부 종료 승인 가능**

---

## 1. Project Overview

### 왜 이 프로젝트를 만들었는가
세탁 프랜차이즈 운영은 단순 주문 관리가 아니라 다음과 같은 복합 흐름을 동시에 다뤄야 합니다.

- 매장 POS 접수와 고객 응대
- 품목 단위 추적과 외부택 관리
- 공장 입고, 공정, 검수, 재작업
- 회차/런/스톱 기반 물류 운영
- 세탁 후 환불, 클레임, 보상 처리
- 본사/지사/매장 간 정책 통제
- 확정 후 직접 수정할 수 없는 정산 구조
- 감사 로그와 승인 워크플로

CleanOps는 이런 운영 흐름을 **실제 기업 시스템 관점**에서 구조화한 설계 프로젝트입니다.

### 목표
- 전국 단위 세탁 프랜차이즈의 운영 시스템을 설계한다.
- HQ / REGION / PLANT / STORE 조직 구조를 하나의 시스템에서 지원한다.
- POS/스캔/물류/정산/클레임 업무를 **OrderItem 중심 추적 구조**로 일관되게 설계한다.
- 정책, 권한, 감사, 정산까지 포함한 **실무형 설계 문서 패키지**를 완성한다.

## 2. Target Organization Structure

```text
HQ
├─ REGION
│  ├─ PLANT
│  └─ STORE
└─ PLANT (본사 직속 공장 가능)
```

### 조직별 역할
| 조직 | 역할 |
|---|---|
| HQ | 정책/가격/정산/권한/감사/전사 통제 |
| REGION | 운영 조정, 물류 관제, 클레임 1차 처리, 정산 검토 |
| PLANT | 공장 입고, 분류, 공정, 검수, 재작업, 출고 준비 |
| STORE | 고객 접점, 접수, 결제, 출고, 환불, 클레임 등록 |

## 3. Core Design Principles

CleanOps는 아래 원칙을 구현 단계에서도 뒤집지 않는 **고정 기준**으로 둡니다.

1. **OrderItem 중심 추적**
   - 주문 헤더보다 품목이 실제 운영 추적 단위입니다.

2. **ExternalTag 1:1 연결**
   - 외부택은 OrderItem과 1:1로 연결되는 물리 식별자입니다.

3. **외부택 자동 할당 + 재사용 금지**
   - 접수 시 품목 수만큼 시스템이 자동 할당하며, 취소/환불/재세탁 이후에도 재사용하지 않습니다.

4. **current_status + status_history 분리**
   - 조회 성능과 감사 추적을 모두 만족시키기 위해 현재 상태와 상태 이력을 분리 관리합니다.

5. **세탁 후 환불 허용 + 통제**
   - 고객 응대 속도를 위해 허용하되, 사유/감사/보고/이상거래 통제를 강하게 둡니다.

6. **REGION 중심 클레임 처리**
   - STORE는 접점, REGION은 1차 처리 중심, HQ는 상위 통제 및 고난도 개입 주체입니다.

7. **정산 확정 후 adjustment-only**
   - 확정 회차는 직접 수정하지 않고 adjustment로만 보정합니다.

8. **RBAC + Scope + State + Approval + Audit**
   - 단순 메뉴 권한이 아니라, 조직 범위·상태·승인·감사까지 포함하는 행위 중심 권한 구조를 사용합니다.

## 4. Core Business Scope

### STORE
- 일반 세탁 / 수선 / 하이엔드 / 재세탁 접수
- 결제 / 부분결제 / 미수금 / 환불
- 출고 / 부분출고 / 출고 증빙
- 고객 CRM / 법인 계정 / 후불/청구 관리

### PLANT
- 입고 스캔
- 분류 / 라우팅 / 배치 편성
- 세탁 / 수선 / 하이엔드 공정 처리
- 검수 / 재작업 / 확인품 / 태그 예외 처리

### LOGISTICS
- 픽업 / 배송 / 인수인계 / 운송 추적
- 회차 / 런 / 스톱 / 기사 작업 관리
- 배송 실패 / 재방문 / 재배송 / 물류 예외 처리

### OUTSOURCING
- 외주 등록
- 반출 / 반입 관리
- 외주 상태 추적
- 외주 비용 / 지연 / 분실 / 불량 관리

### REGION / HQ
- 운영 대시보드
- 정책 승인 / 배포 / 충돌 검증
- 클레임 / 분쟁 / 보상 관리
- 정산 검토 / 확정 / 감사 / 리포트

## 5. Key Operational Flows

### 1) 접수
고객 조회 또는 등록 → 주문 생성 → 품목 입력 → 외부택 자동 할당 → 예상 완료일 계산 → 가격 정책 적용 → 결제 상태 저장 → 주문 확정

### 2) 공장 처리
입고 스캔 → 라우팅 결정 → 공정 실행 → 검수 → 재작업/확인품 분기 → 출고 준비

### 3) 물류
수거 요청 → 인수인계 → 운송 추적 → 매장 도착 → 출고 가능 상태 반영

### 4) 출고
품목 단위 출고 가능 여부 확인 → 부분출고 또는 전체출고 → 출고 증빙 저장

### 5) 환불 / 클레임
환불 요청 → 정책 평가 → 승인/보고/감사 통제 → 환불 완료

클레임 등록 → REGION 검토 → 보상/환불/재세탁/기각 결정 → 필요 시 HQ 에스컬레이션

### 6) 정산
회차 기준 초안 생성 → 검토 → 확정 → 확정 후 이슈는 adjustment로만 반영

## 6. Architecture Summary

### 권장 아키텍처
**Modular Monolith**

이 프로젝트는 주문/품목/외부택/공정/물류/환불/클레임/정산/권한/감사 사이의 정합성과 강한 트랜잭션 요구가 크기 때문에, 1차 구조로는 모듈러 모놀리스를 채택했습니다.

동시에 내부는 도메인 모듈 단위로 분리하여, 향후 일부 영역을 서비스 단위로 분리할 수 있도록 설계했습니다.

### 도메인 모듈
- Organization
- Identity & Authorization
- Customer
- Order
- Logistics
- Plant Operation
- Outsourcing
- Claim & Compensation
- Policy & Pricing
- Settlement
- Common / Audit

### 설계 포인트
- 화면 기준이 아니라 **운영 책임 기준**으로 모듈을 분리
- 상태 전이는 도메인 규칙이 소유
- 본문 저장과 후속 비동기 처리 분리
- 첨부/다운로드/감사/정합성 점검을 공통 통제 계층으로 분리

## 7. Data & Control Design Highlights

### 데이터 모델 핵심
- `orders` : 접수/결제 중심 헤더
- `order_item` : 실질 운영 단위
- `external_tag` : 품목과 1:1 연결되는 물리 식별자
- `status_history` : 상태 전이 원장
- `audit_log` : 중요 변경 감사 원장
- `settlement / settlement_line / settlement_adjustment` : 정산 원장 구조

### 정책 설계 핵심
- 데이터 기반 정책
- HQ > REGION > STORE 우선순위
- 소급 적용 금지
- override는 원정책 직접 수정이 아닌 별도 레이어
- 정책 평가는 화면 조회 시점이 아니라 **업무 이벤트 시점** 기준

### 권한 설계 핵심
- RBAC + Scope + State + Approval + Audit
- 민감 행위(환불, 보상, 정산 확정, 권한 부여, 강제 상태 변경)는 승인/사유/감사 필수
- SoD(직무분리) 적용
